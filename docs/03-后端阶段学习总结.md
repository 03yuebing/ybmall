# ybmall 后端项目阶段学习总结

## 当前学习阶段

从项目初始化、环境配置开始，到目前为止，我们已经完成了商品服务中分类和品牌相关的基础能力学习。

目前学习的重点并不是简单的 CRUD，而是围绕 Spring Boot 后端项目的基础工程结构、接口规范、数据库建模和业务关系展开。

## 1. 多模块项目结构

当前项目采用 Maven 多模块结构。

主要模块包括：

- `ybmall-common`：公共模块，存放多个微服务都可能用到的通用代码。
- `ybmall-product`：商品服务模块，负责商品、分类、品牌、属性等业务。
- 其他模块：如会员、订单、优惠券、库存、网关等，后续学习时逐步展开。

目前我们重点学习的是：

```text
ybmall-common
ybmall-product
```

这种结构的好处是：

- 公共代码可以复用。
- 每个业务服务职责清晰。
- 符合微服务项目的模块拆分方式。

## 2. 统一返回格式 R

项目中所有接口统一返回 `R` 对象。

统一返回格式通常包括：

```json
{
  "code": 0,
  "msg": "success",
  "data": {}
}
```

这样做的意义是：

- 前端可以用统一方式处理接口返回。
- 成功、失败、业务异常都有统一格式。
- 后端接口风格更加规范。

目前 `R` 类放在 `ybmall-common` 模块，因为它属于公共能力，后续其他服务也可以复用。

## 3. 参数校验

我们使用 `@Valid` 配合 VO 对象中的校验注解完成参数校验。

常见校验注解包括：

- `@NotBlank`
- `@NotNull`
- `@Min`
- `@Max`
- `@URL`

参数校验的流程是：

```text
请求进入 Controller
        ↓
@Valid 校验请求参数
        ↓
如果校验失败，抛出 MethodArgumentNotValidException
        ↓
全局异常处理器捕获异常
        ↓
返回统一的 R 结果
```

这样可以避免大量手写 `if` 判断，让参数校验更加规范。

## 4. 全局异常处理

我们实现了全局异常处理器，用来统一处理：

- 参数校验失败
- 业务异常
- 系统未知异常

整体流程可以理解为：

```text
请求进入后端
        ↓
参数校验
        ↓
Controller
        ↓
Service
        ↓
Mapper
        ↓
数据库
```

如果中间任何一步出现异常，就交给全局异常处理器处理，然后返回统一的 `R` 结果。

目前主要涉及：

- `BizCodeEnum`：统一定义错误码和错误信息。
- `BizException`：业务异常类。
- `YbmallExceptionControllerAdvice`：全局异常处理器。

## 5. Entity / VO / Controller / Service / Mapper 分层

目前我们已经接触了后端常见分层结构。

### Entity

Entity 对应数据库表。

例如：

```text
CategoryEntity -> pms_category
BrandEntity    -> pms_brand
```

它主要负责描述数据库中的字段。

### VO

VO 用来接收前端请求参数。

例如：

```text
CategorySaveVo
CategoryUpdateVo
BrandSaveVo
BrandUpdateVo
```

VO 的好处是：

- 不直接暴露数据库实体。
- 可以针对不同接口设计不同参数。
- 可以在 VO 上添加参数校验规则。

### Controller

Controller 负责接收请求，返回响应。

它不应该写复杂业务逻辑，只负责调用 Service。

### Service

Service 负责业务逻辑。

例如：

- 查询分类树
- 保存品牌
- 修改分类
- 删除前检查业务规则

### Mapper

Mapper 负责和数据库交互。

我们使用 MyBatis-Plus 的 `BaseMapper` 来简化基础 CRUD。

## 6. 三级分类表结构与树形结构转换

我们学习了 `pms_category` 表。

这张表使用自关联结构表示三级分类。

核心字段包括：

```text
cat_id      当前分类 id
parent_cid  父分类 id
cat_level   分类层级
```

例如：

```text
家用电器
  └── 电视
      └── 平板电视
```

在数据库中，分类不是嵌套存储的，而是一行一行存储，通过 `parent_cid` 表示父子关系。

Java 中需要把平铺数据转换成树形结构：

```text
List<CategoryEntity> -> Tree
```

核心思路是：

1. 查询所有分类。
2. 按 `parent_cid` 分组。
3. 找到一级分类。
4. 给每个分类设置子分类。
5. 返回树形结构。

这一块比普通 CRUD 更有学习价值，因为它涉及数据结构转换和业务展示结构的组织。

## 7. MyBatis-Plus 分页查询

品牌列表接口中，我们学习了 MyBatis-Plus 的分页查询。

核心对象是：

```java
Page<T>
```

分页查询需要配置分页插件：

```java
MybatisPlusInterceptor
PaginationInnerInterceptor
```

分页接口常见参数包括：

```text
page   当前页
limit  每页条数
key    搜索关键字
```

基本流程是：

```text
前端传入 page、limit、key
        ↓
Service 创建 Page 对象
        ↓
构造 LambdaQueryWrapper
        ↓
调用 MyBatis-Plus 的 page 方法
        ↓
返回分页结果
```

分页查询在真实企业项目中非常常见，尤其是后台管理系统。

## 8. 分类与品牌的多对多关系

我们开始学习了分类和品牌之间的关系。

一个品牌可以属于多个分类：

```text
华为 -> 手机、笔记本、智能手表
```

一个分类也可以拥有多个品牌：

```text
手机 -> 华为、小米、oppo、Apple
```

所以分类和品牌之间是：

```text
多对多关系
```

数据库中通常使用中间表来表示多对多关系。

课程中的表是：

```text
pms_category_brand_relation
```

核心字段包括：

```text
id
brand_id
catelog_id
brand_name
catelog_name
```

其中：

- `brand_id`：品牌 id
- `catelog_id`：分类 id
- `brand_name`：品牌名称
- `catelog_name`：分类名称

注意：课程 SQL 中字段名是 `catelog_id`，不是标准英文的 `catalog_id`。学习时应保持和数据库字段一致。

## 9. 关联表中的冗余字段与数据一致性

`pms_category_brand_relation` 中不仅保存了 id，还保存了名称：

```text
brand_name
catelog_name
```

这两个字段属于冗余字段。

它的好处是：

- 查询方便。
- 展示数据时可以少做表连接。
- 后台页面查询更直接。

它的问题是：

- 主表名称变化后，关系表中的名称可能不同步。

例如：

```text
pms_brand.name = HUAWEI
pms_category_brand_relation.brand_name = 华为
```

这就产生了数据不一致。

所以真实项目中，如果使用冗余字段，就要维护同步逻辑：

```text
修改品牌名称时，同步更新关系表中的 brand_name。
修改分类名称时，同步更新关系表中的 catelog_name。
```

这就是关联表数据一致性维护。

## 当前阶段的理解

到目前为止，我们学习的重点不是“会写几个 CRUD 接口”，而是理解后端项目的基础工程能力：

- 项目如何分模块
- 接口如何统一返回
- 参数如何校验
- 异常如何统一处理
- 数据库表如何映射到 Java 对象
- Controller、Service、Mapper 如何分层
- 树形结构如何从数据库平铺数据转换
- 分页查询如何实现
- 多对多关系如何建模
- 冗余字段如何维护一致性

普通 CRUD 本身并不复杂，真正重要的是 CRUD 背后的工程规范和业务建模能力。
